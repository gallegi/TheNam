# Building Robust ML Pipelines with Python and Docker

One of the hardest problems in applied machine learning is not the modeling — it is the plumbing. Getting data to flow reliably from raw sources through preprocessing, training, evaluation, and deployment requires careful engineering. In this post, I will walk through a practical pattern for building ML pipelines that are reproducible, testable, and easy to extend.

## Why Pipelines Matter

An ad-hoc training script that runs on your laptop is not a pipeline. A pipeline is a system where every stage has a defined input contract, a defined output contract, and can be run independently or as part of a chain. This distinction matters enormously when you need to debug a production issue at 2 AM or hand the project to a new team member.

## Project Structure

Start with a clear directory layout:

```
ml-project/
├── data/
│   ├── raw/
│   └── processed/
├── src/
│   ├── data_ingestion.py
│   ├── preprocessing.py
│   ├── train.py
│   └── evaluate.py
├── configs/
│   └── config.yaml
├── Dockerfile
└── docker-compose.yml
```

Each Python module in `src/` corresponds to one pipeline stage. They communicate through the filesystem (or an artifact store like MLflow) rather than through function calls — this makes each stage independently runnable and testable.

## Dockerizing Each Stage

Docker eliminates the "works on my machine" problem. Here is a minimal `Dockerfile` for a training stage:

```dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY src/ ./src/
COPY configs/ ./configs/

CMD ["python", "src/train.py"]
```

Build and run it with:

```bash
docker build -t ml-trainer .
docker run --rm \
  -v $(pwd)/data:/app/data \
  -v $(pwd)/models:/app/models \
  ml-trainer
```

Mounting volumes keeps data and model artifacts outside the container, so you can iterate on code without losing your outputs.

## Configuration Management

Hard-coding hyperparameters is a common mistake. Instead, use a YAML config file and load it at runtime:

```python
import yaml
from pathlib import Path

def load_config(path: str = "configs/config.yaml") -> dict:
    with open(path) as f:
        return yaml.safe_load(f)

config = load_config()
learning_rate = config["training"]["learning_rate"]
batch_size = config["training"]["batch_size"]
```

This pattern lets you swap configs without touching source code, which is essential for experiment tracking.

## Orchestrating with Docker Compose

For a multi-stage pipeline, `docker-compose.yml` gives you a declarative way to define dependencies:

```yaml
version: "3.9"
services:
  preprocess:
    build: .
    command: python src/preprocessing.py
    volumes:
      - ./data:/app/data

  train:
    build: .
    command: python src/train.py
    volumes:
      - ./data:/app/data
      - ./models:/app/models
    depends_on:
      - preprocess
```

Run the full pipeline with a single command: `docker-compose up`.

## Key Takeaways

Robust ML pipelines share a few common traits: they are containerized for reproducibility, configured externally for flexibility, and structured so each stage can be run and tested in isolation. Start simple, add complexity only when you feel the pain of not having it, and always treat your pipeline code with the same rigor you would apply to application code.
